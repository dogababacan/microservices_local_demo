Async Note (Eventual Consistency)

The HTTP checkout request returns a synchronous 202 Accepted response immediately after the Order Service creates the initial record, confirming the request has been received.

However, downstream validation (such as payment processing or stock reservation) occurs asynchronously via RabbitMQ. If these downstream events fail, compensating transactions are triggered, which asynchronously cancel the order in the background, updating the Order Service record eventually.

Port: 3000 Style: HTTP Entry Endpoint: POST /checkout Calls: Order Service (HTTP)
Owns
System boundary validation, API endpoint definitions, and synchronous client response handling. Generates early correlation IDs.
Does NOT own
Orders database state, stock counts, credit validations, or notification distribution logs.
Why this exists
Web clients need a clean, synchronous HTTP entry point. The Gateway maps this standard HTTP input into event-driven background flows.
Port: 3001 Style: HTTP + RabbitMQ Topic Producer & Consumer Publishes: order.created, order.cancelled Consumes: inventory.failed, payment.failed
Owns
The core order database records and their corresponding lifecycles (created, pending, cancelled, or completed).
Does NOT own
Inventory quantities, catalog availability, or transaction charging operations.
Why these events?
  • order.created: Announces the booking intent.
  • order.cancelled: Cancels the booking to trigger compensations when issues occur downstream.
Port: 3002 Style: RabbitMQ Topic Consumer + Producer Publishes: inventory.reserved, inventory.failed, inventory.released Consumes: order.created, inventory.release_requested
Owns
The catalog counts and physical stock reservations in memory.
Does NOT own
Customer checkout records, transactional flows, or payment confirmations.
Why these events?
  • inventory.reserved: Locks stock for payment validation.
  • inventory.failed: Rejects booking due to zero stock.
  • inventory.released: Restores stock after payment fails (compensation).
Port: 3003 Style: RabbitMQ Topic Consumer + Producer Publishes: payment.completed, payment.failed, inventory.release_requested Consumes: inventory.reserved
Owns
Payment processing logic, merchant authorization states, and simulated card results.
Does NOT own
Product catalogue details, item inventory counts, or client message channels.
Why these events?
  • payment.completed: Notifies order checkout is successful.
  • payment.failed: Reports card declined exceptions.
  • inventory.release_requested: Requests compensation stock restore since payment failed.
Port: 3004 Style: RabbitMQ Topic Consumer Only Consumes: payment.completed, payment.failed, inventory.failed
Owns
Receipt distribution logs, text alerts, and email notifications (customer messages).
Does NOT own
System booking updates, credit transactions, or database inventory balances.
Why this exists
Purely listens to final transaction endpoints to keep customers updated without slowing down order processing.
Port: 3005 Style: RabbitMQ Global Consumer (Wiretap) Consumes: # (All events wild card)
Owns
System-wide performance monitoring metrics, transaction trends, and event logging.
Does NOT own
Transaction execution paths, stock deductions, or client invoicing records.
Why this exists
Demonstrates adding a global monitoring system using topic routing wildcards without requiring code changes in publishers.
AMQP Port: 5672 Management Port: 15672 Exchange: ecommerce_events
Main Role
Decouples microservices. Receives events from publishers and routes copies to queues according to configured binding patterns.
Why topic routing?
Enables precise point-to-point routing, event stream splitting, and global auditing wiretaps within a single centralized broker.
